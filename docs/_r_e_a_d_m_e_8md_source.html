18 BitShares Core is the BitShares blockchain implementation and command-line interface.
19 The web browser based wallet is [BitShares UI](https://github.com/bitshares/bitshares-ui).
20 
21 Visit [BitShares.org](https://bitshares.org/) to learn about BitShares and join the community at [BitSharesTalk.org](https://bitsharestalk.org/).
22 
23 Information for developers can be found in the [BitShares Developer Portal](https://dev.bitshares.works/). Users interested in how BitShares works can go to the [BitShares Documentation](https://how.bitshares.works/) site.

27 ---------------
28 Build instructions and additional documentation are available in the

31 ### Build
32 
33 We recommend building on Ubuntu 16.04 LTS (64-bit)
34 
35 **Build Dependencies:**
36 
37  sudo apt-get update
38  sudo apt-get install autoconf cmake make automake libtool git libboost-all-dev libssl-dev g++ libcurl4-openssl-dev doxygen
39 
40 **Build Script:**
41 
42  git clone https://github.com/bitshares/bitshares-core.git
43  cd bitshares-core
44  git checkout master # may substitute "master" with current release tag
45  git submodule update --init --recursive
46  mkdir build
47  cd build
48  cmake -DCMAKE_BUILD_TYPE=Release ..
49  make
50 
51 **Upgrade Script:** (prepend to the Build Script above if you built a prior release):
52 
53  git remote set-url origin https://github.com/bitshares/bitshares-core.git
54  git checkout master
55  git remote set-head origin --auto
56  git pull
57  git submodule update --init --recursive # this command may fail
58  git submodule sync --recursive
59  git submodule update --init --recursive
60 
61 **NOTE:**
62 
63 * BitShares requires a 64-bit operating system to build, and will not build on a 32-bit OS. Tested operating systems:
64  * Linux (heavily tested with Ubuntu 18.04 LTS and Ubuntu 16.04 LTS)
65  * macOS (various versions)
66  * Windows (various versions, Visual Studio and MinGW)
67  * OpenBSD (various versions)
68 
69 * BitShares requires [Boost](http://www.boost.org/) libraries to build, supports version `1.58` to `1.69`.
70 Newer versions may work, but have not been tested.
71 If your system came pre-installed with a version of Boost libraries that you do not wish to use, you may
72 manually build your preferred version and use it with BitShares by specifying it on the CMake command line.
73 
74  Example: `cmake -DBOOST_ROOT=/path/to/boost ..`
75 
76 * BitShares requires [OpenSSL](https://www.openssl.org/) libraries to build, supports version `1.0.2` to `1.1.1`.
77 If your system came pre-installed with a version of OpenSSL libraries that you do not wish to use, you may
78 manually build your preferred version and use it with BitShares by specifying it on the CMake command line.
79 
80  Example: `cmake -DOPENSSL_ROOT_DIR=/path/to/openssl ..`

83 ### Run the node software
84 
85 **After Building**, the node software `witness_node` can be launched with:
86 
87  ./programs/witness_node/witness_node
88 
89 The node will automatically create a `witness_node_data_dir` directory with some config files.
90 The blockchain data will be stored in the directory too.
91 It may take several hours to fully synchronize the blockchain.
92 
93 You can exit the node using `Ctrl+C`. Please be aware that the node may need some time (usually a few minutes) to exit cleanly, please be patient.
94 
95 **IMPORTANT:** By default the node will start in reduced memory mode by using some of the commands detailed in [Memory reduction for nodes](https://github.com/bitshares/bitshares-core/wiki/Memory-reduction-for-nodes).
96 In order to run a full node with all the account histories (which is usually not necessary) you need to remove `partial-operations` and `max-ops-per-account` from your config file. Please note that currently(2018-10-17) a full node will need more than 160GB of RAM to operate and required memory is growing fast. Consider the following table as **minimal requirements** before running a node:
97 
98 | Default | Full | Minimal | ElasticSearch
99 | --- | --- | --- | ---
100 | 150G HDD, 16G RAM | 640G SSD, 64G RAM * | 120G HDD, 4G RAM | 1TB SSD, 32G RAM
101 
102 \* For this setup, allocate at least 500GB of SSD as swap.
103 
104 To use the command-line wallet or other wallets / clients with the node, the node need to be started with RPC connection enabled, which can be done by starting the node with the `--rpc-endpoint` parameter, E.G.
105 
106  ./programs/witness_node/witness_node --rpc-endpoint=127.0.0.1:8090
107 
108 or configure it in the config file by editing `witness_node_data_dir/config.ini` as follows:
109 
110  rpc-endpoint = 127.0.0.1:8090
111 
112 You can run the program with `--help` parameter to see more info:
113 
114  ./programs/witness_node/witness_node --help

117 ### Run the command-line wallet software
118 
119 To start the command-line wallet, in a separate terminal you can run:
120 
121  ./programs/cli_wallet/cli_wallet
122 
123 **IMPORTANT:** The cli_wallet or API interfaces to the witness node wouldn't be fully functional unless the witness node is fully synchronized with the blockchain. The cli_wallet command `info` will show result `head_block_age` which will tell you how far you are from the live current block of the blockchain.
124 
125 To check your current block:
126 
127  new >>> info
128 
129 To query the blockchain, E.G. get info about an account:
130 
131  new >>> get_account <account_name_or_id>
132 
133 If you need to transact with your account but not only query, firstly set your initial password and unlock the wallet:
134 
135 * For non-Windows operating systems, you can type the commands and press `[ENTER]`, then input the password and press `[ENTER]`, in this case the password won't show:
136 
137  new >>> set_password [ENTER]
138  Enter password:
139  locked >>> unlock [ENTER]
140  Enter password:
141  unlocked >>>
142 
143 * For Windows, or you'd like to show the password, type the commands with the password:
144 
145  new >>> set_password <PASSWORD>
146  locked >>> unlock <PASSWORD>
147  unlocked >>>
148 
149 To be able to transact with your account, import the corresponding private keys:
150 
151  unlocked >>> import_key <ACCOUNT_NAME> <WIF_KEY>
152 
153 The private keys will be encrypted and stored in the wallet file, the file name is `wallet.json` by default.
154 The private keys are accessible when the wallet is unlocked.
155 
156  unlocked >>> dump_private_keys
157 
158 Use `lock` command to make the private keys inaccessible. There is no auto-lock feature so far.
159 
160  unlocked >>> lock
161 
162 To import your initial (genesis) balances, import the private keys corresponding to the balances:
163 
164  unlocked >>> import_balance <ACCOUNT_NAME> [<WIF_KEY> ...] true
165 
166 Use `help` to see all available wallet commands.
167 
168  >>> help
169 
170 Use `gethelp <COMMAND>` to see more info about individual commands. E.G.
171 
172  >>> gethelp get_order_book
173 
174 The definition of all commands is available in the
175 [wallet.hpp](https://github.com/bitshares/bitshares-core/blob/master/libraries/wallet/include/graphene/wallet/wallet.hpp) souce code file.
176 Corresponding documentation can be found in the [Doxygen documentation](https://doxygen.bitshares.org/classgraphene_1_1wallet_1_1wallet__api.html).
177 
178 You can run the program with `--help` parameter to see more info:
179 
180  ./programs/cli_wallet/cli_wallet --help
181 
182 There is also some info in the [Wiki](https://github.com/bitshares/bitshares-core/wiki/CLI-Wallet-Cookbook).

198 -------------
199 
200 ### Node API
201 
202 The `witness_node` software provides several different API's, known as *node API*.
203 
204 Each API has its own ID and a name.
205 When running `witness_node` with RPC connection enabled, initially two API's are available:
206 * API 0 has name *"database"*, it provides read-only access to the database,
207 * API 1 has name *"login"*, it is used to login and gain access to additional, restrictable API's.
208 
209 Here is an example using `wscat` package from `npm` for websockets:
210 
211  $ npm install -g wscat
212  $ wscat -c ws://127.0.0.1:8090
213  > {"id":1, "method":"call", "params":[0,"get_accounts",[["1.2.0"]]]}
214  < {"id":1,"result":[{"id":"1.2.0","annotations":[],"membership_expiration_date":"1969-12-31T23:59:59","registrar":"1.2.0","referrer":"1.2.0","lifetime_referrer":"1.2.0","network_fee_percentage":2000,"lifetime_referrer_fee_percentage":8000,"referrer_rewards_percentage":0,"name":"committee-account","owner":{"weight_threshold":1,"account_auths":[],"key_auths":[],"address_auths":[]},"active":{"weight_threshold":6,"account_auths":[["1.2.5",1],["1.2.6",1],["1.2.7",1],["1.2.8",1],["1.2.9",1],["1.2.10",1],["1.2.11",1],["1.2.12",1],["1.2.13",1],["1.2.14",1]],"key_auths":[],"address_auths":[]},"options":{"memo_key":"GPH1111111111111111111111111111111114T1Anm","voting_account":"1.2.0","num_witness":0,"num_committee":0,"votes":[],"extensions":[]},"statistics":"2.7.0","whitelisting_accounts":[],"blacklisting_accounts":[]}]}

233 ### Wallet API
234 
235 The `cli_wallet` program can also be configured to serve **all of its commands** as API's, known as *wallet API*.
236 
237 Start `cli_wallet` with RPC connection enabled:
238 
239  $ ./programs/cli_wallet/cli_wallet --rpc-endpoint=127.0.0.8091
240 
241 Access the wallet API using an HTTP client:
242 
243  $ curl --data '{"jsonrpc": "2.0", "method": "info", "params": [], "id": 1}' http://127.0.0.1:8091/rpc
244  $ curl --data '{"jsonrpc": "2.0", "method": "get_account", "params": ["1.2.0"], "id": 1}' http://127.0.0.1:8091/rpc
245 
246 Note: The syntax to access wallet API is a bit different than accessing node API.
247 
248 **Important:**
249 * When RPC connection is enabled for `cli_wallet`, sensitive data E.G. private keys which is accessible via commands will be accessible via RPC too. It is recommended that only open network connection to localhost or trusted addresses E.G. configure a firewall.
250 * When using wallet API, sensitive data E.G. the wallet password and private keys is transmitted as plain text, thus may be vulnerable to network sniffing. It is recommended that only use wallet API with localhost, or in a clean network, and / or use `--rpc-tls-endpoint` parameter to only serve wallet API via secure connections.

253 Accessing restrictable node API's
254 ---------------------------------
255 
256 You can restrict node API's to particular users by specifying an `api-access` file in `config.ini`
257 or by using the `--api-access /full/path/to/api-access.json` startup node command. Here is an example `api-access` file which allows
258 user `bytemaster` with password `supersecret` to access four different API's, while allowing any other user to access the three public API's
259 necessary to use the node:
260 
261  {
262  "permission_map" :
263  [
264  [
265  "bytemaster",
266  {
267  "password_hash_b64" : "9e9GF7ooXVb9k4BoSfNIPTelXeGOZ5DrgOYMj94elaY=",
268  "password_salt_b64" : "INDdM6iCi/8=",
269  "allowed_apis" : ["database_api", "network_broadcast_api", "history_api", "network_node_api"]
270  }
271  ],
272  [
273  "*",
274  {
275  "password_hash_b64" : "*",
276  "password_salt_b64" : "*",
277  "allowed_apis" : ["database_api", "network_broadcast_api", "history_api"]
278  }
279  ]
280  ]
281  }
282 
283 Passwords are stored in `base64` as salted `sha256` hashes. A simple Python script,
284 [`saltpass.py`](https://github.com/bitshares/bitshares-core/blob/master/programs/witness_node/saltpass.py)
285 is avaliable to obtain hash and salt values from a password.
286 A single asterisk `"*"` may be specified as username or password hash to accept any value.
287 
288 With the above configuration, here is an example of how to call `add_node` from the `network_node` API:
289 
290  {"id":1, "method":"call", "params":[1,"login",["bytemaster", "supersecret"]]}
291  {"id":2, "method":"call", "params":[1,"network_node",[]]}
292  {"id":3, "method":"call", "params":[2,"add_node",["127.0.0.1:9090"]]}
293 
294 Note, the call to `network_node` is necessary to obtain the correct API identifier for the network API. It is not guaranteed that the network API identifier will always be `2`.
295 
296 The restricted API's are accessible via HTTP too using *basic access authentication*. E.G.
297 
298  $ curl --data '{"jsonrpc": "2.0", "method": "call", "params": ["network_node", "add_node", ["127.0.0.1:9090"]], "id": 1}' http://bytemaster:supersecret@127.0.0.1:8090/
299 
300 Our `doxygen` documentation contains the most up-to-date information
301 about API's for the [node](https://doxygen.bitshares.org/namespacegraphene_1_1app.html) and the
302 [wallet](https://doxygen.bitshares.org/classgraphene_1_1wallet_1_1wallet__api.html).

305 FAQ
306 ---
307 
308 - Is there a way to generate help with parameter names and method descriptions?
309 
310  Yes. Documentation of the code base, including APIs, can be generated using Doxygen. Simply run `doxygen` in this directory.
311 
312  If both Doxygen and perl are available in your build environment, the CLI wallet's `help` and `gethelp`
313  commands will display help generated from the doxygen documentation.
314 
315  If your CLI wallet's `help` command displays descriptions without parameter names like
316  `signed_transaction transfer(string, string, string, string, string, bool)`
317  it means CMake was unable to find Doxygen or perl during configuration. If found, the
318  output should look like this:
319  `signed_transaction transfer(string from, string to, string amount, string asset_symbol, string memo, bool broadcast)`
320 
321 - Is there a way to allow external program to drive `cli_wallet` via websocket, JSONRPC, or HTTP?
322 
323  Yes. External programs may connect to the CLI wallet and make its calls over a websockets API. To do this, run the wallet in
324  server mode, i.e. `cli_wallet -s "127.0.0.1:9999"` and then have the external program connect to it over the specified port
325  (in this example, port 9999). Please check the ["Using the API"](#using-the-api) section for more info.
326 
327 - Is there a way to access methods which require login over HTTP?
328 
329  Yes. Most of the methods can be accessed by specifying the API name instead of an API ID. If an API is protected by a username and a password, it can be accessed by using *basic access authentication*. Please check the ["Accessing restrictable node API's"](#accessing-restrictable-node-apis) section for more info.
330 
331  However, HTTP is not really designed for "server push" notifications, and we would have to figure out a way to queue notifications for a polling client. Websockets solves this problem. If you need to access the stateful methods, use Websockets.
332 
333 - What is the meaning of `a.b.c` numbers?
334 
335  The first number specifies the *space*. Space 1 is for protocol objects, 2 is for implementation objects.
336  Protocol space objects can appear on the wire, for example in the binary form of transactions.
337  Implementation space objects cannot appear on the wire and solely exist for implementation
338  purposes, such as optimization or internal bookkeeping.
339 
340  The second number specifies the *type*. The type of the object determines what fields it has. For a
341  complete list of type ID's, see `GRAPHENE_DEFINE_IDS(protocol, protocol_ids ...)` in
342  [protocol/types.hpp](https://github.com/bitshares/bitshares-core/blob/master/libraries/protocol/include/graphene/protocol/types.hpp)
343  and `GRAPHENE_DEFINE_IDS(chain, implementation_ids ...)` in [chain/types.hpp](https://github.com/bitshares/bitshares-core/blob/master/libraries/chain/include/graphene/chain/types.hpp).
344 
345  The third number specifies the *instance*. The instance of the object is different for each individual
346  object.
347 
348 - The answer to the previous question was really confusing. Can you make it clearer?
349 
350  All account ID's are of the form `1.2.x`. If you were the 9735th account to be registered,
351  your account's ID will be `1.2.9735`. Account `0` is special (it's the "committee account,"
352  which is controlled by the committee members and has a few abilities and restrictions other accounts
353  do not).
354 
355  All asset ID's are of the form `1.3.x`. If you were the 29th asset to be registered,
356  your asset's ID will be `1.3.29`. Asset `0` is special (it's BTS, which is considered the "core asset").
357 
358  The first and second number together identify the kind of thing you're talking about (`1.2` for accounts,
359  `1.3` for assets). The third number identifies the particular thing.
360 
361 - How do I get the `network_add_nodes` command to work? Why is it so complicated?
362 
363  You need to follow the instructions in the ["Accessing restrictable node API's"](#accessing-restrictable-node-apis) section to
364  allow a username/password access to the `network_node` API. Then you need
365  to pass the username/password to the `cli_wallet` on the command line.
366 
367  It's set up this way so that the default configuration is secure even if the RPC port is
368  publicly accessible. It's fine if your `witness_node` allows the general public to query
369  the database or broadcast transactions (in fact, this is how the hosted web UI works). It's
370  less fine if your `witness_node` allows the general public to control which p2p nodes it's
371  connecting to. Therefore the API to add p2p connections needs to be set up with proper access
372  controls.
